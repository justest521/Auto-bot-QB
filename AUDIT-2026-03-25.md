# QB-ERP 系統流程邏輯回測報告
> 日期：2026-03-25
> 範圍：報價→訂單→銷貨→出貨 / 訂單→採購→進貨→入庫 兩條主流程

---

## 🔴 嚴重（會造成資料寫入失敗或資料損壞）

### 1. `convert_order_to_sale` — 訂單直轉銷貨跳過庫存檢查
- **檔案**: `actions-post.js` L1103-1250
- **問題**: 完全不檢查庫存就建立銷貨，且直接把訂單標為 `status: 'completed', shipping_status: 'shipped'`。等於一鍵把「未出貨訂單」標成已出貨，庫存完全不扣。
- **衝突**: 和 `instock_to_sale`（L2452，有庫存檢查的版本）功能重疊且互相矛盾——同一張訂單可能被兩個入口各轉一次。
- **建議**: 廢除 `convert_order_to_sale`，統一走 `instock_to_sale`。

### 2. `convert_order_to_sale` — qb_invoices.order_id 類型不匹配
- **檔案**: `actions-post.js` L1212
- **問題**: `qb_invoices.order_id` 是 **bigint**，但傳入的 `salesLinkId` 可能是 `qb_sales_history.id`（bigint, OK）或 fallback 到 `legacyOrderId`。當 `erp_orders.id`（UUID）無法轉成整數時（L1159 regex），`legacyOrderId = null`，發票的 `order_id` 就是 null，發票和銷貨單斷開。
- **建議**: `qb_invoices.order_id` 改為 text 或新增 `sale_id` 欄位做正確關聯。

### 3. `confirm_stock_in` — reference_id 類型不匹配
- **檔案**: `actions-post.js` L1772-1776
- **問題**: `qb_inventory_movements.reference_id` 是 **bigint**，但傳入的 `stock_in_id` 來自 `erp_stock_ins.id`（**UUID**）。UUID 寫入 bigint 會失敗，但 insert 後沒有 check error，導致庫存異動記錄靜默丟失。
- **同樣問題**: `create_purchase_return`（L1821）傳 UUID `ret.id` 給 bigint `reference_id`。
- **建議**: `qb_inventory_movements.reference_id` 改為 text 類型。

### 4. `create_payment` / `confirm_payment` — order_id 跨表類型不匹配
- **檔案**: `actions-post.js` L1306, L1337-1355
- **問題**: `qb_payments.order_id` 是 **bigint**，但 `create_payment` 做 `Number(order_id)` 把 UUID 轉成 NaN。後續 `confirm_payment` 用 NaN 去查 `erp_orders`（UUID pk），永遠查不到，`payment_status` 永遠不會被更新。
- **建議**: `qb_payments.order_id` 改為 text，或新增 `erp_order_id` UUID 欄位。

### 5. `create_shipment` — erp_shipment_items 缺少 product_id 欄位
- **檔案**: `actions-post.js` L1409
- **問題**: 程式寫入 `product_id: it.product_id`，但 DB `erp_shipment_items` 沒有此欄位。`insertSingleWithColumnFallback` 可能忽略多餘欄位，但行為不可預期。
- **建議**: DB 加 `product_id` 欄位，或程式移除此 field。

## 🟠 中等（流程邏輯缺陷，不會直接炸錯但資料不正確）

### 6. `process_approval` 核准銷貨後不扣庫存、不更新狀態
- **檔案**: `actions-post.js` L2862-2890
- **問題**: `instock_to_sale` 註明「不先扣庫存，等審批通過再扣」（L2487），但 `process_approval` 核准 `doc_type === 'sale'` 時什麼都沒做——既沒扣庫存，也沒更新 `qb_sales_history.status` 從 draft 變 issued。審批通過後銷貨單永遠停在 draft，庫存永遠不扣。
- **建議**: 在 `process_approval` 加入 sale 核准邏輯：扣庫存 + 更新 status 為 issued。

### 7. `convert_order_to_sale` 不寫 source_id，導致關聯查詢斷開
- **檔案**: `actions-get.js` L889-893 / `actions-post.js` L1103-1250
- **問題**: GET `order_items_with_stock` 用 `source_id = order_id` 查關聯銷貨。`instock_to_sale` 有寫 `source_id`，但 `convert_order_to_sale` 沒寫，所以從舊入口轉的銷貨在訂單詳情頁看不到。
- **建議**: 統一入口或補上 source_id。

### 8. `consolidate_orders_to_po` 的 linked_pos 查詢失敗
- **檔案**: `actions-get.js` L896-904 / `actions-post.js` L2096-2164
- **問題**: GET 用 `ilike remark %order_no%` 查關聯 PO。`shortage_to_po` 的 remark 含 order_no 能 match，但 `consolidate_orders_to_po` 的 remark 是 `彙整自 N 筆訂單`，完全不含 order_no，合併採購單會查不到。
- **建議**: PO 改用 FK `source_order_id` 做正確關聯，別靠 remark 字串比對。

### 9. `confirm_order_with_stock` 允許庫存負數
- **檔案**: `actions-post.js` L2198-2203
- **問題**: 明確允許 `newStock = currentStock - orderQty` 為負值。但 `instock_to_sale` 要求 `stockQty >= orderQty` 才能轉銷貨。先 confirm 扣成負數後，同張訂單再想轉銷貨就永遠卡住。
- **建議**: confirm 時不應直接扣庫存（那是銷貨確認才做的事），或至少不允許負數。

### 10. 出貨不需要銷貨單
- **檔案**: `actions-post.js` L1362-1468
- **問題**: `create_shipment` 可直接用 `order_id` 建出貨單，不檢查有無銷貨單。正常流程應是「訂單→銷貨→出貨」，但系統允許跳過銷貨。
- **建議**: 加入前置檢查：該訂單是否已有對應銷貨單（status=issued 以上）。

## 🟡 分頁 Bug（16 個 endpoint 第一頁漏資料）

以下 API 全部使用 `page default '0'` + `offset = page * limit`，但前端 Pager 元件起始頁是 1（disabled={page <= 1}）。前端傳 page=1 時 offset = 1 × limit，直接跳過第一頁：

| # | API endpoint | 行號 |
|---|-------------|------|
| 1 | products | L1203 |
| 2 | chat_history | L1264 |
| 3 | line_customer_tags | L1414 |
| 4 | chat_history_stats | L1464 |
| 5 | inventory | L1486 |
| 6 | shipments | L1529 |
| 7 | returns | L1644 |
| 8 | inquiries | L1740 |
| 9 | stock_ins | L1811 |
| 10 | purchase_returns | L1834 |
| 11 | vendor_payments | L1851 |
| 12 | stocktakes | L1875 |
| 13 | stock_adjustments | L1896 |
| 14 | reorder_suggestions | L2155 |
| 15 | invoices | L2208 |
| 16 | approvals | L2259 |

**已修好的**: orders, quotes, purchase_orders, sales_documents, formal_customers（用 `page || '1'` + `(page-1)*limit`）

**建議**: 統一改為 `page default '1'` + `offset = (page - 1) * limit`。

## 🟡 其他問題

### 12. `confirm_stock_in` 和 `confirm_stock_in_with_inventory` 功能重複
- **檔案**: `actions-post.js` L1757 vs L2247
- **問題**: 兩個 handler 做同樣的事（確認進貨+加庫存），但 `confirm_stock_in` 額外寫 `qb_inventory_movements`（有 reference_id 類型問題），`confirm_stock_in_with_inventory` 不寫異動記錄但會清 reorder_suggestions。
- **建議**: 合併為一個，保留最完整的邏輯。

### 13. 退貨核准 `approve_return` 不建庫存異動記錄
- **檔案**: `actions-post.js` L1587-1596
- **問題**: 直接改 `quickbuy_products.stock_qty` 但沒 insert `qb_inventory_movements`，庫存帳對不起來。
- **建議**: 加入 inventory_movement 記錄。

### 14. `create_purchase_return` 建立時就扣庫存
- **檔案**: `actions-post.js` L1812-1824
- **問題**: 建立進貨退出（status=pending）時就扣庫存。應等核准後才做庫存動作（和銷退 approve_return 邏輯一致）。
- **建議**: 把扣庫存邏輯移到審批通過後。

### 15. `consolidate_orders_to_po` 缺少 subtotal/tax_amount
- **檔案**: `actions-post.js` L2129-2136
- **問題**: 建 PO 時只寫 `total_amount`，沒寫 `subtotal` 和 `tax_amount`，和 `create_purchase_order`（L1700-1705）不一致。
- **建議**: 補上 subtotal 和 tax_amount 計算。

## 修復優先順序建議

1. **Phase 1 — DB schema 修正** (Migration)
   - `qb_inventory_movements.reference_id` → text
   - `qb_payments` 加 `erp_order_id` UUID 欄位（或 order_id 改 text）
   - `qb_invoices.order_id` → text（或加 sale_id）
   - `erp_shipment_items` 加 `product_id` uuid 欄位
   - `erp_purchase_orders` 加 `source_order_ids` text[] 欄位

2. **Phase 2 — 重複入口清理**
   - 廢除 `convert_order_to_sale`，統一走 `instock_to_sale`
   - 合併 `confirm_stock_in` 和 `confirm_stock_in_with_inventory`

3. **Phase 3 — 審批流程補完**
   - `process_approval` 加入 sale 核准：扣庫存 + 更新 status
   - `process_approval` 加入 stock_in 核准邏輯（如需要）
   - `create_purchase_return` 扣庫存移到審批後

4. **Phase 4 — 分頁統一修正**
   - 16 個 endpoint 統一改 `(page-1)*limit`

5. **Phase 5 — 其他修正**
   - approve_return 加庫存異動記錄
   - consolidate_orders_to_po 補 subtotal/tax
   - confirm_order_with_stock 移除直接扣庫存或禁止負數
   - create_shipment 加銷貨單前置檢查
