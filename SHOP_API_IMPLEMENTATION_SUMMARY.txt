================================================================================
DATABASE INTEGRATION
================================================================================

Primary Table (Read):
  - quickbuy_products
    Columns: id, item_number, description, tw_retail_price, tw_reseller_price,
             us_price, product_status, category, image_url, weight_kg,
             origin_country, commodity_code, search_text, safety_stock,
             replacement_model, created_at, updated_at

Write Tables (Primary):
  - erp_orders (customer_name, customer_phone, customer_email, customer_line_id,
                note, source, status, created_at, ...)
  - erp_order_items (order_id, product_id, quantity)

Fallback Tables (if Primary doesn't exist):
  - shop_orders (same schema as erp_orders)
  - shop_order_items (same schema as erp_order_items)

Query Method: Supabase REST API via fetch()
Auth: Service role key (SUPABASE_SERVICE_KEY)

PERFORMANCE OPTIMIZATIONS

✓ Supabase REST API (optimized queries)
✓ Count queries use GROUP BY for efficiency
✓ Rate limiting prevents resource exhaustion
✓ Force-dynamic rendering for real-time data
✓ Minimal response payloads

Cache Strategy:
  - Fresh cache (< 5min):    X-Cache: HIT
  - Expired but working:      X-Cache: MISS
  - Fallback on DB error:     X-Cache: STALE
  - Error with no cache:      X-Cache: ERROR + error response

QUERY EXAMPLES

================================================================================
ERROR HANDLING
================================================================================

Products Endpoint:
  - 400: Invalid query parameters
  - 429: Rate limit exceeded
  - 500: Database query failed

Categories Endpoint:
  - 429: Rate limit exceeded
  - 500: Database error (with fallback to stale cache)

Order Endpoint:
  - 400: Missing required fields, invalid email, product not found
  - 429: Rate limit exceeded
  - 500: Database write error

All errors return JSON with descriptive error messages.

================================================================================
RATE LIMITING DETAILS
================================================================================

Implementation: In-memory store per IP address
Cleanup: Old entries removed every 5 minutes
Persistence: Per-function instance (Vercel serverless compatible)

Limits:
  /api/shop/products:   60 requests/60 seconds per IP
  /api/shop/categories: 60 requests/60 seconds per IP
  /api/shop/order:      30 requests/60 seconds per IP

Rate Limit Headers:
  X-RateLimit-Limit:     Maximum requests allowed
  X-RateLimit-Remaining: Requests remaining in window
  X-RateLimit-Reset:     Unix timestamp when limit resets
  Retry-After:           Seconds to wait before retry
